Functional-Light-JS

2020-11-19The total word count - 606Estimated time of reading - 3 mins

最近看functional-light-JS这本书到递归的时候，发现看不懂了。想起B站上有作者跟书同名的课程，就想对比和总结一下从视频和从书本上，学到的东西。

觉得书本讲的东西更多，但视频讲的更容易懂，以下基本是从视频中记到的一些内容。

Imperative vs. Declarative

The former focuses more on “how to do that”.

While the latter one focuses more on “why we do this?”

Comments are always explaining why we do this way but not what the code is doing.

Like, “Why we add one but not two or three to this number?”

Functional Programing does jobs of comment by code itself.

Once you know 1+1=2, you can benefit from it and do any sumation.In other sense, once you follow the principles of FP, you will benefit from it and no need to prove it.

(Yep, you may not know how to prove 1+1=2, right?)(No bb, just follow.)

f(x) = 2(x^3)+1

FP is not always about the function keyword.

A function call, always receive paramters, do something with the params, and finally output the results.

Functions that don’t return anything are not functions but procedure.”Procedure Programming” rather than “Functional Programming”.

A procedure used in a function, can contaminate the function, resulting the two things both become procedures.

Pure function:

  1. Same input, same output.(not strictly)
  2. A function does nothing with any variables outside of the scope of the function.(same as the above,not strictly enough)

Do you think the following addAnother is a pure function?

It can be, though it accesses z, which is in the global scope.

Directly looking at the function definition, we may easily consider it a impure function because of the variable z.

We have to mentally run all the code before the function call to judge whether the function is pure.

And, what about re-defining the definition of addTwo? Can you still assert that addAnother is pure?

If you are sure that any variables accessed outside of the function scope are immutable throughout all the programme, you can say that this function will not make mistakes.*Are you sure about that? *

Pure function is judged throughout the piece of programming.Maybe in the function scope, or the block, or even the global scope.It all depends.

Side effects are almost impossible to be eliminated, but we can reduce them, make them obvious and then, traceable.

Two ways to reduce side effect / make them obvious.

  1. Extract the impurtiy.

    Side effects poluting the global scope is bad, but you can wrap them in another scope, so that they can affect as fewer as possible.(like, wrapping them in a function).

  2. Containing the impurity

    Make copies of the variables, do with the copied ones, and restore the ones that are passed in.

    (You can the output, and nothing outside of the scope is changed.)
